## 一、RAG 系统优化

### 1.1 检索质量

| # | 严重度 | 问题 | 位置 | 修复方案 |
|---|--------|------|------|----------|
| R1 | **P0** | 向量检索分数伪造——混合检索中使用 `1.0 - (i/len)` 替代 Chroma 返回的真实 distance | `hybrid_search.py:140-142` | 从 Chroma 响应中提取 distances，用 `1/(1+distance)` 转为相似度 |
| R2 | **P0** | 双层检索每次都重建临时 BM25 索引——O(n) per query | `two_layer_search.py:164-169` | 维护持久摘要 BM25 索引，仅在文档变更时重建 |
| R3 | **P0** | 嵌入文本过滤空格导致维度不匹配——filtered_texts 可能少于输入 | `embedding.py:91-104` | 在调用端处理索引对齐，不在 provider 内过滤 |
| R4 | P1 | `relevance_threshold=0.6` 已定义但从未使用——低相关度摘要仍进入二层 | `two_layer_search.py:24` | 在 `_search_summaries` 返回前添加分数过滤 |
| R5 | P1 | RRF 融合用文本前100字符作为文档 ID——可能冲突合并不同 chunk | `hybrid_search.py:163-168` | 在 metadata 中添加唯一 `chunk_id` |
| R6 | P1 | `final_select_count=3` 过于保守——每轮只向 LLM 提供 3 个 chunk | `config.py:104` | 调整为 5-8 |
| R7 | P1 | 流式与非流式回答使用完全不同的 prompt 模板——同问题不同答案 | `qa.py:227-264` vs `qa.py:565-657` | 统一 prompt 模板，或至少在结构上保持一致 |

## 二、技能调用机制优化

### 2.1 严重 Bug

| # | 严重度 | 问题 | 位置 | 修复方案 |
|---|--------|------|------|----------|
| S1 | **P0** | `should_use_skill()` 在同一请求中被调用两次——每次都是完整 LLM API 调用（~2000 tokens），浪费成本且两次结果可能不一致 | `routes.py:454` + `qa.py:412` | routes.py 的判断结果通过参数传递给 `stream_answer()`，避免重复调用 |
| S2 | **P0** | `SkillSelector` 中 Ollama 模型名硬编码 `qwen3:4b-instruct`——用户配置无效 | `skill_selector.py:68` | 改为 `self.cfg.get("ollama_model")` |
| S3 | **P0** | Shell 脚本参数传递断裂——`count` 仅通过环境变量传递但脚本不读取 | `skill_executor.py:219-223` | 所有 params 作为位置参数传递，或脚本改为读取环境变量 |
| S4 | P1 | 技能失败后无 RAG 降级、无重试——`use_skill=true` 时 `docs=[]`，skill 失败则 context 为空 | `qa.py:423-439` | 添加 fallback：skill 失败 → 自动走 RAG 检索 |

### 2.2 参数传递缺陷

| # | 严重度 | 问题 | 位置 | 修复方案 |
|---|--------|------|------|----------|
| S5 | P1 | `SkillSelector` 的参数名映射和补全仅在 ReAct 模式生效——非 ReAct 模式缺少 `_validate_and_complete_params` | `react_agent.py:235-290` vs `qa.py:429` | 将补全逻辑提升到 `SkillExecutor` 或 `SkillManager` 层 |
| S6 | P1 | 天气技能参数名不一致——LLM 输出 `location` 但代码只检查 `city` → 回退到"宁波" | `react_agent.py:284-286` | 统一参数名映射：`location` → `city` |
| S7 | P2 | `SkillSelector` 输出 Schema 硬编码只有 `query` 和 `location`——新技能无法自定义参数 | `skill_selector.py:167-170` | 根据技能 metadata 动态生成参数 schema |

### 2.3 技能触发方式改进方案

当前仅支持自然语言触发（通过 LLM 判断），建议实现三层体系：

| 层级 | 触发方式 | LLM 成本 | 延迟 | 准确率 | 实现难度 |
|------|----------|----------|------|--------|----------|
| **Layer 1** | 命令式 `/skill` | 0 | 极低 | 100% | 低（正则匹配） |
| **Layer 2** | 自然语言（当前） | ~2000 tokens | +0.5-2s | 85-95% | 已实现 |
| **Layer 3** | ReAct 推理 | ~5000+ tokens | +3-10s | 95%+ | 已实现 |

**推荐**：命令式 + 自然语言 + (可选)ReAct 三层共存，用户自由选择。
