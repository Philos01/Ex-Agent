# 图检索和父子检索架构升级优化计划

## 概述

本次优化目标是实现更智能的检索流程：**图检索 → LLM 逻辑判断 → 父子检索（带线索）**

---

## 核心设计理念

### 判断标准：逻辑优先，而非数量优先

- ❌ **旧方式**：看有多少个实体、多少个文档（数量判定）
- ✅ **新方式**：看回答问题所需的逻辑链路是否完整（逻辑判定）

---

## 总体架构

```
用户问题
    ↓
[图检索] → 结构化信息（实体、关系、文档名称）
    ↓
[LLM 逻辑判断]
    ├─ 判断标准：逻辑链路是否完整？
    ├─ sufficient=true → [只用图检索结果]
    └─ sufficient=false → [提取线索] → [父子检索（使用线索）]
    ↓
[构建 Context]
    ↓
[LLM 生成回答]
```

## 详细实施计划

### 第一步：实现 LLM 逻辑判断模块

#### 新建文件
`backend/app/services/graph_retrieval_judge.py`

#### 核心函数

##### 1. `judge_graph_sufficiency(question: str, route_result: Dict) -> Dict`

**功能说明：**
- 用 LLM 判断图检索结果是否足够回答问题
- 如果不够，给出父子检索线索

**参数说明：**
- `question`: 用户的问题
- `route_result`: 图检索的结果

**返回格式：**
```json
{
    "sufficient": true/false,
    "reason": "简短的判断原因",
    "suggestions": {
        "target_documents": ["文档1.md", "文档2.md"],
        "focus_entities": ["实体1", "实体2"],
        "search_angle": "建议的检索角度",
        "query_hint": "优化后的查询词"
    }
}
```

---

##### 2. `_rule_based_judge(question: str, route_result: Dict) -> Dict`

**功能说明：**
- LLM 调用失败时的规则 fallback

**规则设计：**
- 规则 1：实体列表查询且有 2+ 实体 → sufficient=true
- 规则 2：有关系路径或有 2+ 文档 → sufficient=true
- 规则 3：其他情况 → sufficient=false，suggestions 用图找到的文档和实体

---

#### 提示词设计

##### JUDGE_SYSTEM_PROMPT

```

核心判断原则：

分两步判断：
1. 先分析：回答用户的问题，需要哪些关键信息点？
2. 再检查：图检索的结果是否覆盖了这些关键信息点？

  - 且直接关系路径完整

- sufficient=false 的情况：
  - 问题需要详细描述/解释/总结
  - 问题需要具体数据/参数/数值

- search_angle: 建议的检索角度（如：详细工作信息、技术参数、项目背景等）
- query_hint: 建议的检索查询（可以优化原问题）

只输出 JSON，不要其他内容。
```

### 第二步：增强父子检索，支持线索传递

#### 修改文件
`backend/app/services/parent_retriever.py`

#### 新增方法

##### 1. `search_in_documents(self, query: str, target_documents: List[str]) -> List[Dict]`

**功能说明：**
- 在指定文档内进行父子检索

**实现逻辑：**
```
1. 初始化 vector store collection
2. 遍历 target_documents 中的每个文档
3. 对每个文档：
   a. 用 query 在该文档内搜索（where={"source": doc_name}）
   b. 获取 top 10 个 child chunks
4. 合并所有文档的 child results
5. 用这些 child chunks 查找对应的 parent documents
6. 返回 parent documents 列表
```

---

##### 2. `enhance_query(self, original_query: str, suggestions: Dict) -> str`

**功能说明：**
- 综合利用线索优化查询

**实现逻辑：**
```
1. 从 suggestions 提取：
   - query_hint
   - focus_entities
   - search_angle
2. 构建 parts 列表：[query_hint]
3. 如果有 focus_entities，加入 parts
4. 如果有 search_angle，加入 parts
5. 去重（小写比对）
6. 用空格连接，返回最终查询
```

---

##### 3. `rerank_with_focus(self, results: List[Dict], focus_entities: List[str]) -> List[Dict]`

**功能说明：**
- 根据关注的实体进行重排序

**实现逻辑：**
```
1. 如果没有 focus_entities，直接返回原结果
2. 遍历每个 result：
   a. 把 text 转小写
   b. 计算提到了多少个 focus_entities
   c. 给 result 增加 focus_match_count 字段
   d. 计算 adjusted_score = original_score + (match_count * 0.2)
3. 按 adjusted_score 降序排序
4. 返回排序后的结果
```

---

##### 4. `_deduplicate_results(self, results: List[Dict]) -> List[Dict]`

**功能说明：**
- 去重结果

**实现逻辑：**
```
1. 用 source 或 parent_id 作为 key 去重
2. 保留分数较高的那个
3. 返回去重后的结果
```

---

##### 5. `retrieve_with_clues(self, original_query: str, suggestions: Dict) -> List[Dict]`

**功能说明：**
- 综合利用所有线索的主检索方法

**实现逻辑：**
```
1. 从 suggestions 提取所有线索
2. 用 enhance_query 优化查询
3. 两阶段检索：
   a. 阶段 1：如果有 target_documents，先在这些文档内搜索
   b. 阶段 2：如果结果 < 3，做全库检索
4. 去重
5. rerank_with_focus 重排序
6. 只保留前 N 个（parent_max_count）
7. 返回结果
```

### 第三步：修改主流程，集成新架构

#### 修改文件
`backend/app/services/qa/stream_answer.py`

#### 修改点 1：图检索后的判断逻辑

**修改位置：** 在 `# ── Skill selection ─────────────────` 之后

**新逻辑：**
```python
# 原有图检索代码，保留获取 _graph_result

# 新增：LLM 逻辑判断
_graph_context = ""
_use_parent = False
_parent_suggestions = {}

if _graph_result:
    # 调用 LLM 判断
    from app.services.graph_retrieval_judge import judge_graph_sufficiency
    judge_result = judge_graph_sufficiency(question, _graph_result)
    
    if judge_result["sufficient"]:
        # 只用图检索结果
        _graph_context = _format_graph_context(_graph_result)
        logger.info("图检索逻辑链路完整，只用图检索")
    else:
        # 图不够，准备用父子检索
        _use_parent = True
        _parent_suggestions = judge_result["suggestions"]
        # 同时保留图的上下文作为补充
        _graph_context = _format_graph_context(_graph_result)
        logger.info("图检索逻辑不完整，使用父子检索，线索: {_parent_suggestions}")
```

---

#### 修改点 2：Context 构建

**修改位置：** 在获取检索结果后构建 Context 的部分

**新逻辑：**
```python
# 修改 context 构建部分
context = ""

if _use_parent:
    # 使用父子检索 + 线索
    from app.services.parent_retriever import get_parent_retriever
    parent_retriever = get_parent_retriever()
    parent_docs = parent_retriever.retrieve_with_clues(question, _parent_suggestions)
    # 构建父子检索的 context
    parent_context = _format_parent_documents(parent_docs)
    context = _graph_context + "\n\n---\n\n" + parent_context
else:
    # 原有逻辑（混合/向量检索）
    # ...
```

## 线索使用详解

| 线索 | 用途 | 实现在哪个方法 |
|-----|------|--------------|
| `target_documents` | 限制检索范围，只在这些文档内搜索 | `search_in_documents` |
| `focus_entities` | 1. 查询优化时加入<br>2. 结果重排序时加分 | `enhance_query` + `rerank_with_focus` |
| `search_angle` | 查询优化时加入 | `enhance_query` |
| `query_hint` | 替代/补充原查询，提高检索质量 | `enhance_query` |

## 示例场景

### 场景 1：事实查询（图足够）

```
用户问题："张三在哪个公司工作？"

图检索结果：
- 相关实体：["张三", "宁波拓普"]
- 关系路径：["张三-工作单位-宁波拓普"]

LLM 判断过程：
1. 需要的关键信息点：张三实体、工作单位关系、公司名称
2. 图检索覆盖了全部信息点
3. 判断：sufficient=true

结果：只用图检索回答
```

---

### 场景 2：需要详细信息（图不够）

```
用户问题："详细介绍一下张三的背景？"

图检索结果：
- 相关实体：["张三"]

LLM 判断过程：
1. 需要的关键信息点：教育背景、工作经历、项目经验等
2. 图检索只找到张三这个实体，信息点严重不足
3. 判断：sufficient=false

线索生成：
- target_documents: [图检索找到的相关文档]
- focus_entities: ["张三"]
- search_angle: "详细个人背景介绍"
- query_hint: "张三 背景介绍 教育 工作经历"

结果：使用父子检索补充信息
```

---

### 场景 3：开放性问题（图不够）

```
用户问题："关于这个项目你能告诉我什么？"

图检索结果：
- 相关实体：["项目A", "Python", "TensorFlow"]
- 相关文档：["项目介绍.md"]

LLM 判断过程：
1. 需要的关键信息点：项目背景、目标、技术栈、团队等
2. 图检索只找到少量实体和文档名称
3. 判断：sufficient=false

线索生成：
- target_documents: ["项目介绍.md"]
- focus_entities: ["项目A", "Python", "TensorFlow"]
- search_angle: "项目完整介绍"
- query_hint: "项目A 介绍 背景 目标 技术栈"

结果：使用父子检索补充信息
```

1. **Phase 1**：先写 `graph_retrieval_judge.py`（LLM 逻辑判断）

## 风险点和处理

| 风险 | 影响 | 处理方案 |
|-----|------|---------|
| LLM 判断超时或失败 | 阻塞流程 | 增加规则 fallback |
| LLM 判断错误 | 结果质量下降 | 日志记录，人工可干预 |
| 父子检索线索处理复杂 | 开发周期长 | 先实现核心功能，再迭代优化 |
